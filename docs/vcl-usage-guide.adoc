== Overview

VCL (VeriSim Consonance Language) is the query layer that Hypatia uses to interrogate
security scan data, pattern registries, recipes, outcomes, and dispatch logs.
Hypatia's VCL implementation supports two execution backends: a *FileExecutor*
that reads from the `verisim-data` Git repository's flat files, and an
*ArangoDB* backend for graph traversal queries.

== Architecture

The VCL pipeline flows through the following stages:

[source,text]
----
                          +-------------------+
                          |   VCL Client      |
                          |   (GenServer)     |
                          |   lib/vcl/        |
                          |   client.ex       |
                          +--------+----------+
                                   |
                             VCL string
                                   |
                          +--------v----------+
                          |     Parser        |
                          |  (built into      |
                          |   Client)         |
                          +--------+----------+
                                   |
                                  AST
                                   |
                     +-------------+-------------+
                     |                           |
            +--------v----------+       +--------v----------+
            |   FileExecutor    |       |    ArangoDB        |
            | lib/vcl/          |       | lib/data/          |
            | file_executor.ex  |       | arangodb.ex        |
            +-------------------+       +--------------------+
                     |                           |
            verisim-data/             ArangoDB graph
            (flat JSON files)           (trust, confidence,
                                         anomalies)
----

=== Component Summary

[cols="1,2,1"]
|===
| Component | Role | Location

| VCL Client
| GenServer with built-in parser, query cache (60s TTL), and stats tracking
| `lib/vcl/client.ex`

| FileExecutor
| Maps VCL store names to `verisim-data` directory paths and reads flat files
| `lib/vcl/file_executor.ex`

| Query
| 20+ high-level query functions built on top of the VCL Client
| `lib/vcl/query.ex`

| VerisimdbConnector
| VCL-powered connector with file I/O fallback
| `lib/verisimdb_connector.ex`

| ArangoDB
| Graph traversal, trust propagation, and confidence history
| `lib/data/arangodb.ex`
|===

== VCL Client (`lib/vcl/client.ex`)

The VCL Client is an Elixir GenServer that manages query execution. It provides:

* **Built-in VCL parser** -- parses VCL query strings into an internal AST
* **Query cache** -- caches results with a 60-second TTL; a GenServer timer
  automatically evicts expired entries
* **Stats tracking** -- counts queries executed, cache hits, and errors
* **Pluggable executors** -- dispatches parsed ASTs to FileExecutor or ArangoDB

=== Starting the Client

The VCL Client starts as part of the Hypatia application supervision tree
(see `lib/application.ex`). It can also be started manually:

[source,elixir]
----
{:ok, pid} = Hypatia.VCL.Client.start_link([])
----

=== Executing a Raw Query

[source,elixir]
----
{:ok, results} = Hypatia.VCL.Client.query("FROM scans SELECT * WHERE severity = 'high'")
----

== FileExecutor (`lib/vcl/file_executor.ex`)

The FileExecutor maps VCL store names to directories within the local
`verisim-data` Git repository clone.

=== Store-to-Directory Mapping

[cols="1,2,2"]
|===
| VCL Store | Directory Path | File Format

| `scans`
| `verisim-data/scans/`
| One JSON file per repository (e.g., `echidna.json`)

| `patterns`
| `verisim-data/patterns/registry.json`
| Single JSON registry file

| `recipes`
| `verisim-data/recipes/`
| Recipe files named `recipe-*.json`

| `outcomes`
| `verisim-data/outcomes/`
| JSONL files (one JSON object per line)

| `dispatch`
| `verisim-data/dispatch/`
| JSONL files (one JSON object per line)

| `index`
| `verisim-data/index.json`
| Single JSON index file
|===

=== How FileExecutor Works

1. The VCL Client parses the query string into an AST.
2. The AST's `FROM` clause identifies the store name.
3. FileExecutor resolves the store name to a filesystem path.
4. FileExecutor reads and decodes the JSON/JSONL files.
5. `WHERE` clause filters, `SELECT` projections, and `ORDER BY`/`LIMIT` are
   applied in-memory.
6. Results are returned to the VCL Client, which caches them.

== Query Module (`lib/vcl/query.ex`)

The Query module provides 20+ high-level functions that compose VCL queries
internally. These functions are the primary interface for Hypatia's other
modules (pattern analyzer, recipe matcher, fleet dispatcher, etc.).

=== Core Query Functions

[source,elixir]
----
# Fetch all scans
{:ok, scans} = Hypatia.VCL.Query.fetch_scans()

# Fetch a single repository's scan
{:ok, scan} = Hypatia.VCL.Query.fetch_scan("echidna")

# Fetch the pattern registry
{:ok, patterns} = Hypatia.VCL.Query.fetch_patterns()

# Fetch all recipes
{:ok, recipes} = Hypatia.VCL.Query.fetch_recipes()

# Fetch outcomes
{:ok, outcomes} = Hypatia.VCL.Query.fetch_outcomes()

# Fetch the global index
{:ok, index} = Hypatia.VCL.Query.fetch_index()

# Fetch substitution rules
{:ok, subs} = Hypatia.VCL.Query.fetch_substitutions()
----

=== Filtered Query Functions

[source,elixir]
----
# Filter scans by severity level
{:ok, high_scans} = Hypatia.VCL.Query.scans_by_severity("high")

# Filter scans by category
{:ok, injection_scans} = Hypatia.VCL.Query.scans_by_category("injection")

# Filter recipes by tier
{:ok, tier1} = Hypatia.VCL.Query.recipes_by_tier(1)

# Filter recipes by language
{:ok, elixir_recipes} = Hypatia.VCL.Query.recipes_by_language("elixir")

# Filter recipes above a confidence threshold
{:ok, confident} = Hypatia.VCL.Query.recipes_above_confidence(0.85)
----

== VerisimdbConnector (`lib/verisimdb_connector.ex`)

The VerisimdbConnector is the higher-level integration point. It wraps the VCL
Client and Query module, adding:

* **VCL-first execution** -- all queries go through VCL Client by default
* **File I/O fallback** -- if the VCL Client process is unavailable (crashed,
  not started), the connector falls back to direct file reads using
  `File.read!/1` and `Jason.decode!/1`, logging a warning via `Logger.warning/1`

=== Fallback Behavior

[source,elixir]
----
# Normal path (VCL Client running):
#   VerisimdbConnector -> VCL.Client -> FileExecutor -> verisim-data/scans/
#
# Fallback path (VCL Client unavailable):
#   VerisimdbConnector -> File.read! -> Jason.decode!
#   Logger.warning("VCL Client unavailable, falling back to direct file read")
----

== Query Examples

=== Example 1: Find All High-Severity Findings

[source,elixir]
----
# Using the Query module (recommended)
{:ok, results} = Hypatia.VCL.Query.scans_by_severity("high")

# Using raw VCL
{:ok, results} = Hypatia.VCL.Client.query(
  "FROM scans SELECT * WHERE severity = 'high' ORDER BY repo ASC"
)
----

[source,elixir]

  %{"repo" => "verisim", "severity" => "high", "category" => "config", ...}
]
----

== Cache Behavior

The VCL Client maintains an in-process query cache:

* **TTL**: 60 seconds per entry
* **Eviction**: A `Process.send_after/3` timer fires every 60 seconds,
  triggering `handle_info(:evict_cache, state)` which removes expired entries
* **Cache key**: The raw VCL query string (after normalization)
* **Cache hit**: Returns cached result without hitting FileExecutor
* **Cache miss**: Executes the query, stores the result with a timestamp
* **Manual invalidation**: Not currently exposed (cache is time-based only)

The cache is particularly useful for the analytics functions
(`pipeline_health/0`, `cross_repo_patterns/0`, etc.) which aggregate multiple
stores and would otherwise re-read several files per call.

== Fallback Behavior

If the VCL Client GenServer is not running or not responding:

1. The VerisimdbConnector detects the missing process.
2. It logs a warning: `Logger.warning("VCL Client unavailable, falling back to direct file read")`.
3. It reads the relevant `verisim-data/` files directly using `File.read!/1`.
4. It decodes with `Jason.decode!/1`.
5. It returns results in the same format as the VCL path.

This ensures Hypatia remains functional even if the VCL subsystem has issues,
at the cost of losing caching and structured query capabilities.
